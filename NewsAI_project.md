## 二、商业模式画布

### 客户细分

- **核心客户**：中型科技公司、AI 创业公司、内容驱动的 to C 品牌
- **典型场景**：5–50 人规模的内容/市场团队，需要每天产出 AI 相关内容（产品博客、官方公众号、行业速递、内部知识库）
- **次级场景**：高校研究室、个人 AI Builder、AI 媒体团队

### 价值主张

> "把硅谷的 AI 热点，连夜翻译成小红书的爆款。"

企业内容团队的真实痛点：
1. **信息过载**：arXiv / Reddit / GitHub / X / HN 每天产出几百条 AI 信息，人工筛选不可持续
2. **跨语境转化难**：英文技术内容直接翻译成中文 → 没人看；要会本地化的"爆点改写"
3. **多平台分发碎**：同一个选题，小红书要标题党 + emoji，公众号要深度长文，B 站要短视频脚本
4. **复盘断层**：内容发完就发完，数据回流不到选题决策

NewsAI 用 Agent 虚拟组织一次性解决以上四个断点，**所有产出沉淀在多维表格里，企业可以直接接管、二次编辑、复用沉淀**。

### 渠道通路

- **首发渠道**：飞书生态内（飞书应用市场 / 多维表格模板分享）
- **拓展渠道**：作为飞书企业方案的 AI 内容场景案例

### 关键资源 / 壁垒

- **多 Agent 编排经验**：基于 LangGraph 的可扩展虚拟组织框架
- **跨语境内容转译方法论**：硅谷信息源 × 中文爆款基因的 prompt 工程沉淀
- **飞书 Base 深度集成**：把 Base 当数据底座、看板、协作面板三合一使用

### 成本结构

- **大模型 Token**：单条完整内容生产约消耗 15k–30k token（豆包 1.6 同等定价测算）
- **图像生成**：每条内容平均 1–3 张配图，按即梦 API 计费
- **飞书侧**：商业版用户免费使用 OpenAPI

## 三、用户洞察

### 目标用户画像

**主要用户**：AI 公司的内容运营经理（30 岁左右，1–3 人内容小团队 leader）

她每天的真实工作流：
- 早 9 点：刷 Twitter / Reddit / 公众号，从一堆信息里挑今天的选题
- 上午：写 1–2 条小红书 / 公众号草稿
- 下午：找设计师配图、找视频同事剪短视频
- 晚上：看昨天发的内容数据，决定明天选题方向

她的核心痛点：
1. "我每天 80% 时间在**筛选和复述别人的信息**，不是在创作"
2. "英文圈火的东西，直接翻译过来中文圈不买账，要重新包装一遍才行"
3. "数据复盘没人做，老板问起来只能拍脑袋说'嗯这个选题反响不错'"

### 真实场景还原

> "周一早上 9 点，运营经理打开飞书多维表格的 NewsAI 工作空间。
> 
> 她看到 `选题库` 表里 Agent 已经为她准备了 5 条今日推荐选题，每条都标注了来源（"今早 Reddit r/LocalLLaMA 热帖 + 上周小红书爆款标题结构"），以及预估爆点（"开发者社区 vs 普通用户的认知差"）。
> 
> 她勾选其中 2 条，点击"启动生产"。半小时后，`内容稿件` 表里出现了 6 份成品（每条选题 × 3 个平台版本：小红书 / 公众号 / B 站短视频脚本），全部带配图说明和 prompt。
> 
> 她在 `审查记录` 表里调整了一个用词，点确认。`分发计划` 表自动生成了今天 + 明天的发布时间表。
> 
> 她回家。第二天早上，`数据回流` 表里出现了昨天发布内容的数据，`选题库` 表里今日推荐已经基于昨天数据自动调整了方向。"

## 四、产品方案

### 4.1 产品定位

**NewsAI = 一个跑在飞书多维表格上的虚拟内容编辑部**。

它不是 SaaS，不是飞书插件，而是 **"一套 Agent 员工 + 一套预置的多维表格业务系统"** —— 部署后企业立刻拥有一个 7×24 工作的 AI 内容团队。

### 4.2 虚拟组织成员（Agent 花名册）

| 花名 | 角色 | 职责 | 主要协作方 |
|---|---|---|---|
| **小哨** TrendScout | 信息官 | 全球信息源监控，抓取潜在选题原料 | → 分析师 |
| **小析** HookAnalyst | 爆点分析师 | 拆解热帖为什么火，提炼可复用钩子 | → 总编 |
| **小编** TopicCurator | 选题总编 | 结合人设/爆点/信息，产出今日选题 | ← 分析师，→ 创作组 |
| **小文** ContentWriter | 文字编辑 | 写正文（多平台版本） | ← 总编，→ 审核 |
| **小图** VisualDesigner | 视觉设计 | 配图说明 + 文生图 prompt + 文字卡片 | ← 总编，→ 审核 |
| **小播** ScriptWriter | 短视频编剧 | 30–60 秒短视频脚本（钩子+核心+CTA） | ← 总编，→ 审核 |
| **小审** Reviewer | 审核员 | 事实核查、风险词、人设一致性 | ← 创作组，→ 分发 |
| **小发** Distributor | 分发策略 | 平台×时间×文案变体规划 | ← 审核 |
| **小数** Analyst | 数据分析师 | 复盘报告 + 反馈到选题方向 | ← 数据回流，→ 总编 |

> 9 位员工，3 个核心岗（分析/创作/复盘），覆盖完整内容生命周期。

### 4.3 业务流程

```
全球信息源 (arXiv/Reddit/HN/GitHub) + 爆款样本 (小红书/抖音 mock)
                            │
                  [小哨] 信息源采集
                            │
                  [小析] 爆点拆解 (并发 N 角度)
                            │
                  [小编] 选题生成 (结合 KOC人设)
                            │
       ┌────────────────────┼────────────────────┐
   [小文] 文字            [小图] 视觉           [小播] 视频脚本
       └────────────────────┼────────────────────┘
                            │
                  [小审] 内容审查
                            │
                  [小发] 分发策略
                            │
                       (mock 发布)
                            │
                  [小数] 数据复盘 ──→ 反馈回 [小编]
```

### 4.4 飞书多维表格作为业务系统

每个业务环节都对应一张 Base 表，**所有 Agent 通过 lark-oapi SDK 真实读写 Base**：

| 表 | 内容 | 读写方 |
|---|---|---|
| `信息源配置` | 监控的平台列表 + 关键词 | 小哨 读 |
| `热帖池` | 抓取到的原始素材 | 小哨 写，小析 读 |
| `爆点洞察` | 多角度爆点拆解结果 | 小析 写，小编 读 |
| `KOC 人设` | 品牌定位、风格、受众（**手动维护**） | 小编/小文/小图/小播 读 |
| `选题库` | 待执行选题 + 状态字段 | 小编 写，KOC 选，创作组 读 |
| `内容稿件` | 多平台版本成品 | 小文/小图/小播 写，小审 读 |
| `审查记录` | 审核流水 + 修改建议 | 小审 写 |
| `分发计划` | 平台×时间×版本 | 小发 写 |
| `数据回流` | 发布后表现（mock） | 小数 读 |
| `Agent 花名册` | 员工档案 + 工作日志 | 所有 Agent 写 |

**多维表格视图玩法**：
- `选题库` 看板视图：按状态分列（推荐 / 已选 / 进行中 / 已完成 / 已弃）
- `内容稿件` 分组视图：按平台分组（小红书 / 公众号 / B 站）
- `Agent 花名册` 仪表盘：每个 Agent 本周产出量、平均耗时、成功率

### 4.5 差异化创新

1. **跨语境信息套利**：硅谷源 × 中文爆款基因，是其他内容工具不做的事
2. **Agent 真员工感**：花名册 + 工作日志 + 协作关系，让虚拟组织"看得见"
3. **Base 即业务系统**：不是 Base 当数据库用，而是 Base 当工作台用，企业可以直接接管编辑
4. **数据回流闭环**：发布数据真实反馈到选题决策，不是一次性内容工厂

## 五、AI 原生设计

### 5.1 AI 核心能力

| 能力 | 用在哪 | 实现 |
|---|---|---|
| 文本理解 + 摘要 | 小哨抓取后初步分类、小析拆解爆点 | LLM (Doubao 1.6) |
| 跨语境改写 | 小编从英文源 → 中文选题，小文写中文正文 | LLM + Prompt 工程 |
| 多角度并发分析 | 小析同时跑 3–5 个不同分析视角 | LangGraph fan-out |
| 多模态生成 | 小图产出文字卡片图 (HTML 渲染) + 信息图 (SVG) + 画面图 (即梦 API) | 模板 + LLM + 即梦 |
| 工具调用 | 所有 Agent 调用飞书 Base 读写 | LangChain Tools 包装 lark-oapi |
| 状态化编排 | 7 阶段状态图 + 中间产出沉淀 | LangGraph StateGraph |

### 5.2 痛点解决映射

| 用户痛点 | AI 解决方案 |
|---|---|
| 信息过载 | 小哨 7×24 监控 + 小析自动拆解爆点 |
| 跨语境难 | 小编内置"中文爆款基因库"，懂小红书/抖音的语言 |
| 多平台分发碎 | 小文/小图/小播 一次产出多平台版本 |
| 复盘断层 | 小数自动分析数据并反馈到选题决策 |

### 5.3 技术架构

```
┌────────────────────────────────────────────┐
│  LangGraph StateGraph                       │
│  ├─ Nodes: 9 Agents                         │
│  ├─ Fan-out: TrendScout / HookAnalyst /     │
│  │            ContentCreator (3-way)        │
│  └─ Shared State: 流转中的全部业务上下文     │
└────────────────────────────────────────────┘
            │                  │
            ▼                  ▼
   LLM Client            Lark Tools
   (OpenAI 协议           (lark-oapi SDK
    指向火山方舟)          封装为 LangChain
                          Tools)
            │                  │
            ▼                  ▼
       豆包 2.0           飞书多维表格 Base
                          (10 张业务表)
```

## 六、落地与商业

### 6.1 技术可行性

- ✅ **无技术黑盒**：LangGraph、lark-oapi、Doubao API 全部成熟
- ✅ **国内合规**：模型走豆包，图走即梦，全部国内服务
- ✅ **一键复现**：bootstrap.py 自动建表 + 配置 + 跑通端到端 Demo

### 6.2 盈利模式

**B 端订阅 + 用量计费**：
- **基础版**：¥199/月/团队，含 5000 token 池，3 个 Agent
- **专业版**：¥999/月/团队，含 50000 token 池，全 9 个 Agent
- **企业版**：年单 ¥30000+，私有化部署 + 定制 Agent + 飞书企业版集成

### 6.3 市场价值

- **TAM**：中国内容驱动型企业 100w+，AI 工具渗透率 < 10%
- **进入策略**：从飞书生态切入（飞书企业用户 1000w+），先做内容场景标杆案例
- **复用空间**：同一套虚拟组织框架可复用到招聘、客服、电商运营场景

## 八、附录：差异化卖点 Pitch

> 大多数 AI 内容工具在做：「写一篇文章」。
> 
> NewsAI 在做：「**让一个 AI 编辑部全天候为你工作**」。
> 
> 它不是给你一个聊天框，而是给你一支 9 人的虚拟团队，跑在你的飞书多维表格里，从全球信息源到中文爆款，从选题到分发到复盘，每个环节都看得见、可干预、可接管。
> 
> 这是 AI 时代企业内容生产的应有形态。
