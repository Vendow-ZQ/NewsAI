# Documents Design · v2

> v2 架构的 4 类内容产物设计（Bitable-only 版本）。
>
> **注意**：v3.0 已升级为 Bitable-Centric 混合架构 —— 状态与元数据存 Bitable，长文本产物存飞书云文档，Bitable 留存文档链接。本文件保留 v2 历史决策记录，v3 产物形态详见代码实现。
>
> v2 核心理念（历史）：**所有产物全部存储在 Bitable 多维表格中，不创建任何飞书云文档**。表既是骨架（控制流转），也是肉（具体内容）。

---

## v2 关键变更

| 维度 | v1 | v2 原方案 | v2 终版 (Bitable-only) |
|---|---|---|---|
| 内容产物形态 | 多维表格的多行文本字段 | 飞书云文档（独立文件 + 分享链接） | **Bitable 多行文本字段** |
| 审改记录 | 表的多行文本字段 | 累积式审改文档 | **TOPIC.审改记录字段（多行文本，累积追加）** |
| 经验沉淀 | 无（数据回流只是表） | 经验文档（小数月度产出） | **DATA.经验总结字段（多行文本）** |
| 飞书 API 依赖 | 仅 Bitable | Bitable + Docs + Drive | **仅 Bitable** |
| 权限复杂度 | 低 | 高（需文档创建 + 权限设置） | **低（仅表格权限）** |
| 评审环境兼容 | 100% | 文档权限可能受限 | **100%** |

## 4 类内容产物总览（Bitable 存储）

| # | 产物类型 | 创建者 | 修改者 | 读者 | 存储位置 | 字段类型 |
|---|---|---|---|---|---|---|
| 1 | **帖子内容** | 小文 ContentWriter | 小改 Editor（在副本上修改后回写） | 小审、小改、小发 | TOPIC.帖子内容 | 多行文本 |
| 2 | **视频脚本内容** | 小播 ScriptWriter | 小改 Editor（在副本上修改后回写） | 小审、小改、小发 | TOPIC.视频脚本内容 | 多行文本 |

### 3.4 累积式追加机制

**关键规则**：
- 每一轮审查 + 修改 = 在字段内容末尾追加 2 个章节（"## 第 N 轮审查" + "## 第 N 轮修改"）
- **不删除历史轮次**——评委能看到完整审改演进，加分项
- 通过后自动追加"## 终稿"章节
- 卡死（达到 max_revisions 仍未通过）追加"## 终止" + "需人工介入" 章节

**技术实现**：
```python
# 伪代码
def review_loop(topic_id, max_revisions=3):
    revision = 0
    audit_record = ""  # TOPIC.审改记录字段内容
    
    while revision < max_revisions:
        revision += 1
        verdict = reviewer.review(topic_id, revision)
        audit_record += f"\n\n## 第 {revision} 轮审查\n{verdict.report}"
        
        if verdict.passed:
            audit_record += f"\n\n## 终稿\n{verdict.final_version}"
            update_topic_field(topic_id, "审改记录", audit_record)
            update_topic_status(topic_id, "审改通过")
            return
        
        edit = editor.revise(topic_id, verdict.feedback, revision)
        audit_record += f"\n\n## 第 {revision} 轮修改\n{edit.changelog}"
        update_topic_field(topic_id, "审改记录", audit_record)
    
    # 达到上限仍未通过
    audit_record += "\n\n## 终止 - 需人工介入\n..."
    update_topic_field(topic_id, "审改记录", audit_record)
    update_topic_status(topic_id, "卡死")
```

### 3.5 状态判定

- **进行中**：审改记录字段包含"第 N 轮审查"但无"终稿"章节
- **已通过**：审改记录字段包含"终稿"章节
- **卡死**：审改记录字段包含"终止"章节

```python
from feishu_adapter.base import FeishuBaseManager

def update_topic_content(base_manager: FeishuBaseManager, topic_id: str, content: str):
    """更新选题的帖子内容字段"""
    base_manager.update_record(
        table_name="选题库",
        record_id=topic_id,
        fields={
            "帖子内容": content,  # 多行文本字段，直接写 Markdown 字符串
            "状态": "生产中"
        }
    )

def append_audit_record(base_manager: FeishuBaseManager, topic_id: str, new_section: str):
    """追加审改记录（读取现有内容 + 追加新章节）"""
    # 1. 读取现有记录
    record = base_manager.get_record("选题库", topic_id)
    current_audit = record.fields.get("审改记录", "")
    
    # 2. 追加新内容
    updated_audit = current_audit + "\n\n" + new_section
    
    # 3. 写回
    base_manager.update_record(
        table_name="选题库",
        record_id=topic_id,
        fields={"审改记录": updated_audit}
    )
```

## 通用约定

### 1. Markdown 格式标准

**所有多行文本字段使用标准 Markdown 语法**：
- `#` H1 标题、`##` H2 标题、`###` H3 标题
- `-` 无序列表、`1.` 有序列表
- `**粗体**`、`*斜体*`
- `` `行内代码` ``、`` ```代码块 ``` ``
- `---` 分隔线
- `|` 表格（简化版）

**飞书 Bitable 渲染效果**：
- 多行文本字段在飞书 Bitable 中会自动渲染 Markdown 格式
- 评委点击字段即可看到格式化后的内容
- 支持折叠/展开，长内容不占用表格行高

### 2. 内容版本管理

| 场景 | 处理方式 |
|---|---|
| 小文一次过 | 帖子内容字段 = 终稿 |
| 小审通过（无需修改） | 审改记录字段 = 审查通过备注 |
| 小审需修改 | 审改记录字段 = v0原稿 + 第1轮审查 + 第1轮修改 + ... + 终稿 |
| 卡死 | 审改记录字段 = 完整记录 + 终止声明 |

### 3. 权限管理

| 字段 | 创建者 | 修改者 | 查看者 |
|---|---|---|---|
| 帖子内容 | 小文 | 小发（发布后固化） | 所有 Agent + KOC |
| 视频脚本内容 | 小播 | 小发（发布后固化） | 所有 Agent + KOC |
| 审改记录 | 小审 | 小审、小改（追加） | 所有 Agent + KOC |
| 经验总结 | 小数 | 小数（追加） | 小编（重点）、其他 Agent |

### 4. 生命周期

- **创建后不删除**：所有字段内容作为产物沉淀，永久保留
- **每条选题都有完整产物链**：帖子内容 + 视频脚本内容 + 审改记录
- **审改记录累积**：从 v0 到终稿的完整演进历史

## 架构决策记录（Bitable-only）

### 决策 1：为什么放弃飞书云文档？

**原方案（表 + 文档）的问题**：
1. **API 权限复杂**：飞书文档需要 `docx:document` + `drive:drive` 权限，评审环境可能受限
2. **Block 转换成本高**：Markdown → 飞书 Block 需要复杂转换器，MVP 阶段投入产出比低
3. **链接管理复杂**：需要维护 record_id → document_token → share_url 的多层映射
4. **评审兼容性**：评委可能没有飞书文档访问权限，但一定有表格权限

**Bitable-only 的优势**：
1. **权限简单**：仅 Bitable 权限，评审环境 100% 兼容
2. **零转换成本**：Markdown 直接写入多行文本字段，飞书自动渲染
3. **原生体验**：评委在 Bitable 里看内容，无需跳转到文档
4. **开发速度快**：省去文档创建、Block 转换、权限设置、链接回填等环节

### 决策 2：4 类文档 → 4 个字段，是否损失结构化？

**权衡**：
- **损失**：无法像飞书文档那样精细控制格式（如特定段落评论、@提及）
- **获得**：开发速度提升 3 倍，评审兼容性 100%，内容 100% 可追溯

**MVP 阶段选择**：接受结构化损失，换取速度和兼容性。

### 决策 3：多行文本字段长度是否足够？

**计算**：
- 公众号 3000 字 + 小红书 500 字 + 抖音 200 字 + B站 800 字 = 约 4500 字
- 审改记录（3 轮）= 约 3000 字
- 视频脚本 = 约 1500 字
- **总计**：约 9000 字 = 约 30,000 字符
- **Bitable 限制**：100,000 字符
- **结论**：足够，且有 3 倍余量。
