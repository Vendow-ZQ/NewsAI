# NewsAI · Project Brief · v2

> **一句话定义**：跑在飞书 Bitable 多维表格上的 AI 新闻编辑部 —— 一组虚拟员工 7×24 把全球 AI 信息源转译为中文爆款内容，全程产物可追溯、可接管、可学习。**所有飞书调用基于 Bitable，不涉及飞书云文档。**

---

## v2 关键升级（vs v1）

| 维度 | v1 | v2 | 升级原因 |
|---|---|---|---|
| **产物形态** | 全部塞进 Base 字段 | **Bitable 多行文本字段（全部产物）** | Base 字段装不下 3000 字稿件曾是顾虑，但飞书文档 API 权限复杂、block 转换成本高、评审环境兼容性差。v2 终版：所有产物存在 Bitable 多行文本字段，零云文档依赖。 |
| **审改机制** | 无（线性流程）| **审-改循环（最多 3 轮）** | 真实编辑部都有审改机制 |
| **Agent 阵容** | 9 个（含小析）| 9 个（删小析、加小改）| 职责合并 + 专人修改 |
| **数据闭环** | 数据回流就是表 | 数据回流 + 月度经验文档 | 系统能"学习" |
| **组织架构** | 3 部门 | 4 部门 + 1 独立 | 加治理组，增加可视化 |

## 二、商业模式画布

### 客户细分

- **核心客户**：中型科技公司、AI 创业公司、内容驱动的 to C 品牌
- **典型场景**：5–50 人规模的内容/市场团队，需要每天产出 AI 相关内容
- **次级场景**：高校研究室、个人 AI Builder、AI 媒体团队

### 价值主张

> "把硅谷的 AI 热点，连夜翻译成小红书的爆款。**所有产物都在飞书生态里，团队即接即用。**"

企业内容团队的真实痛点：

| 痛点 | NewsAI v2 怎么解 |
|---|---|
| **信息过载** —— 每天 100+ 条 AI 信息无法人工筛 | 小哨多源监控 + 主题标签自动打标 |
| **跨语境失败** —— 英文圈火的直接翻译没人看 | 小编内置中文爆款基因库，跨语境改写 |
| **多平台分发碎** —— 一条选题要写 4 个平台版本 | 小文一次产出 4 平台版本，全部沉淀到 Bitable 多行文本字段 |
| **品控失序** —— 内容发出去出错难收回 | **小审-小改循环审改机制**，多轮把关后再分发 |
| **复盘断层** —— 数据反馈不到下次决策 | 小数月度产出经验文档，自动反馈到小编选题 |

### 核心差异化

**vs 通用内容工具（Notion AI / 文心一言等）**：
- 通用工具是"帮你写一篇"——NewsAI 是"一支 AI 编辑部为你 7×24 工作"

**vs 国外类似产品（Cohere / Jasper）**：
- 国外工具不懂中文爆款——NewsAI 内置"硅谷信息源 × 中文爆款基因"

**vs 飞书生态内其他模板**：
- 一般是"用 AI 帮你填表"——NewsAI 是"虚拟组织真的在协同工作"，含完整产物（表 + 文档 + 数据闭环）

- **首发渠道**：飞书生态（飞书应用市场 / 多维表格模板分享 / 飞书企业方案案例）

## 四、产品方案

### 4.1 产品定位

**NewsAI = 跑在飞书生态上的 AI 内容编辑部**。

它不是 SaaS、不是飞书插件、不是 Chatbot，而是 **"一套 Agent 员工 + 一套预置的飞书 Bitable"** —— 部署后企业立即拥有一个可见可干预的 AI 内容团队。

### 4.2 核心创新

#### 创新 1：Bitable-only 架构（核心准则）

**准则**：所有飞书调用基于 Bitable 多维表格，不涉及飞书云文档。

**v1 的问题**：所有产出塞进 Base 字段，3000 字公众号稿件挤在多行文本里，体验差。

**v2 原方案的陷阱**：曾设计"表 + 文档双形态"，用飞书云文档存储 4 类内容产物。但飞书文档 API 权限复杂、block 转换成本高、评审环境兼容性差。

**v2 终版解法**：
- **Base 负责一切**（7 张表）：业务流转 + 状态 + 关联 + 内容存储
- **4 类文档 → 4 个多行文本字段**：帖子内容、视频脚本、审改记录、经验总结全部存在 Bitable 表中
- **无飞书云文档依赖**：零文档创建权限要求，零 block 格式转换，评审环境 100% 兼容

**字段映射**：
| 原文档类型 | 现存储位置 | 字段类型 |
|-----------|-----------|---------|
| 帖子文档 | TOPIC.帖子内容 | 多行文本 |
| 视频脚本文档 | TOPIC.视频脚本内容 | 多行文本 |
| 审改文档 | TOPIC.审改记录 | 多行文本（累积追加）|
| 经验文档 | DATA.经验总结 | 多行文本 |

**评委体验**：打开 `选题库` 看到全流程状态 + 点字段查看完整内容产物，**飞书 Bitable 原生体验**。

#### 创新 2：审改循环

```
小文/小图/小播 写完 → 一次过，不再改
        ↓
小审 创建审改文档 + 复制 v0 原稿
        ↓
小审 写第 1 轮审查意见（追加到审改文档）
        ↓
小改 读意见 + 修改（追加到审改文档）
        ↓
小审 写第 2 轮审查 → 通过 OR 继续
        ↓
最多 3 轮，通过后追加"## 终稿"
```

**关键点**：
- **生产组不污染**：小文/小图/小播 一次过原则，专注创作
- **审改可追溯**：审改文档累积式追加，评委能看到完整演进
- **修改专人化**：小改是独立 Agent，与小审形成专业治理闭环

#### 创新 3：4 部门虚拟组织

```
信息组 → 决策组 → 生产组（并发）→ 治理组（循环）
                                                ↓
                                  独立复盘 → 反馈到决策
```

每个部门有明确职责、leader、汇报关系。**评委打开 `Agent花名册` 表看到的是真实的虚拟公司架构**。

#### 创新 4：跨语境信息套利

> "硅谷 AI 热点 × 中文爆款基因 = 内容护城河"

- **信息源**：4 真（arXiv / HN / GitHub / Reddit）+ 3 mock（小红书 / 抖音 / X）
- **真实价值**：英文 AI 圈每天产出几十条增量信息，中文 KOC 不可能跟上
- **小编核心能力**：从英文源抽取信息 → 用中文爆款语言重新包装

### 4.3 完整业务流程图

```
                    [SRC 信源配置]
                          ↓
              [小哨 TrendScout · 信息组]
                          ↓
                    [TREND 热帖库]
                          ↓
              [小编 TopicCurator · 决策组]
              （吸收原小析的爆点拆解）
                          ↓
                    [TOPIC 选题库]
                          ↓
                    KOC 选定
                          ↓
       ┌──────────────────┴──────────────────┐
   [小文 ContentWriter] [小图 VisualDesigner] [小播 ScriptWriter]
            ↓                  ↓                    ↓
      [帖子文档]           [配图嵌入]          [视频脚本文档]
            └──────────────────┬──────────────────┘
                               ↓
                ┌──────────────┴──────────────┐
                ↓                              ↓
       [小审 Reviewer]   ←────  循环  ────→   [小改 Editor]
       创建审改文档                            修改副本
       追加审查章节                            追加修改章节
                ↓
            通过 OR 卡死
                ↓
              [小发 Distributor]
              更新 TOPIC.分发计划 JSON
                ↓

                ↓

### 4.4 飞书集成深度

**Base 集成**（lark-oapi SDK）：
- 7 张表（4 业务 + 3 元数据）
- 看板视图（按状态分列、按部门分组）
- 仪表盘（OKR 完成度、Agent 工时分布）

**文档集成**（lark-oapi SDK）：
- 4 类文档自动创建到 `NewsAI产物/` 文件夹
- markdown → 飞书 block 转换器（自研）
- 分享链接自动回填到 Base

**权限集成**：
- bootstrap.py 自动配置评委账号权限
- 应用机器人写入 + 用户编辑

## 五、AI 原生设计

### 5.1 AI 核心能力清单

| 能力 | 用在哪 | 实现 |
|---|---|---|
| 多源信息采集 | 小哨 | 4 真爬虫 + 3 mock 数据 |
| 多角度爆点拆解 | 小编 | LangGraph fan-out，5 个分析角度 |
| 跨语境改写 | 小编、小文 | LLM + 中文爆款 prompt 工程 |
| 4 平台多版本生成 | 小文、小播 | 一次 LLM 调用产出 4 平台版本 |
| 多模态生成 | 小图 | 文字卡片（HTML+Playwright）+ 信息图（SVG）+ 文生图（即梦）|
| 审改循环判定 | 小审 | LLM 输出结构化 verdict（pass/needs_revision）|
| 数据闭环分析 | 小数 | 数据汇总 + LLM 洞察生成 |
| 工具调用 | 所有 Agent | LangChain Tools 包装 lark-oapi |
| 状态化编排 | 全流程 | LangGraph StateGraph + 循环边 |

### 5.2 痛点 → AI 解决映射

| 用户痛点 | AI 解决方案 | 用户感知 |
|---|---|---|
| 信息过载 | 小哨 7×24 监控 + 自动打标 | "Base 里只剩高质量素材" |
| 跨语境失败 | 小编中文爆款基因库 | "选题描述就像爆款标题" |
| 多平台版本碎 | 小文一次产 4 版 | "一份文档看到 4 个版本" |
| 品控失序 | 小审-小改循环 | "审改文档能看到完整演进" |
| 复盘断层 | 小数月度经验文档 | "下次选题质量自动提升" |

### 5.3 技术架构

```
┌────────────────────────────────────────────────────┐
│  LangGraph StateGraph                               │
│                                                      │
│  Nodes: 9 个 Agent                                  │
│  ├─ Linear: 小哨 → 小编                             │
