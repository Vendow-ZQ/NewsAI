# Agent 花名册 · Schema & 种子数据

> 这份文档定义了 Agent 花名册表（员工档案）和 Agent 协作日志表（工作日报）的字段 schema，以及 9 位虚拟员工的完整种子数据。
> 
> **角色**：Agent 花名册**不参与主流程**，是展示用 + 元数据用。评委打开飞书 Base 看到这张表，立刻 get 到"这是一个虚拟组织"。

---

## 一、两张表的关系

```
┌─────────────────────────┐         ┌────────────────────────────┐
│   Agent 花名册            │  1:N   │   Agent 协作日志             │
│   (EMP-xxx)             │ <───── │   (LOG-xxxxxxxx-xxx)         │
│                         │         │                              │
│   员工的"身份档案"         │         │   员工的"日常工作流水"        │
│   静态、低频更新           │         │   动态、每次任务执行后追加     │
└─────────────────────────┘         └────────────────────────────┘
```

- **花名册**：每位员工一条记录（共 9 条），相当于"员工身份卡"
- **协作日志**：每次 Agent 跑一次任务追加一条，相当于"日报流水"

---

## 二、Agent 花名册表 Schema

### 2.1 表前缀
- 表名：`Agent花名册`
- ID 前缀：`EMP`（例：`EMP-001`）

### 2.2 字段定义（飞书名片风）

| 字段名 | 类型 | 必填 | 备注 |
|---|---|---|---|
| `id` | 文本 | ✅ | 主键，格式 `EMP-001` |
| `花名` | 文本 | ✅ | 中文昵称（小哨） |
| `英文代号` | 文本 | ✅ | 英文 ID（TrendScout） |
| `职位` | 文本 | ✅ | 一句话 title |
| `头像 emoji` | 文本 | ✅ | 用 emoji 当头像，飞书风格 |
| `一句话职责` | 长文本 | ✅ | 名片上的 tagline |
| `主要职能` | 多选 | ✅ | 能力标签 |
| `直接协作方` | 多选 | ✅ | 与哪些 Agent 共事 |
| `汇报对象` | 单选 |  | 上下游关系（一般是下游 Agent） |
| `本月 OKR` | 长文本 | ✅ | O + 3 个 KR |
| `OKR 完成度` | 数字 | ✅ | 0-100，本月进度 |
| `本周产出概览` | 长文本 | ✅ | 数字化成果 |
| `状态` | 单选 | ✅ | 在岗 / 待命 / 任务中 / 异常 |
| `入职时间` | 日期 | ✅ | 加入日期 |
| `所属部门` | 单选 | ✅ | 内容采集部 / 内容创作部 / 内容运营部 |

### 2.3 部门划分

把 9 位 Agent 编入 3 个虚拟部门，让组织结构感更强：

```
┌──────────────────────────────────────────────────────────┐
│  内容采集部（Intelligence Dept.）                          │
│  └─ 小哨 TrendScout（信息官）                              │
│  └─ 小析 HookAnalyst（爆点分析师）                          │
│                                                            │
│  内容创作部（Creation Dept.）                              │
│  └─ 小编 TopicCurator（选题总编）⭐ 部门 leader            │
│  └─ 小文 ContentWriter（文字编辑）                          │
│  └─ 小图 VisualDesigner（视觉设计师）                       │
│  └─ 小播 ScriptWriter（短视频编剧）                         │
│                                                            │
│  内容运营部（Operations Dept.）                            │
│  └─ 小审 Reviewer（审核员）                                │
│  └─ 小发 Distributor（分发策略师）                          │
│  └─ 小数 Analyst（数据分析师）                              │
└──────────────────────────────────────────────────────────┘
```

## 四、Agent 协作日志表 Schema（工作日报）

### 4.1 表前缀
- 表名：`Agent协作日志`
- ID 前缀：`LOG`（例：`LOG-20260503-001`）

### 4.2 字段定义

| 字段名 | 类型 | 必填 | 备注 |
|---|---|---|---|
| `id` | 文本 | ✅ | `LOG-{YYYYMMDD}-{NNN}` |
| `员工 ID` | 文本 | ✅ | 引用 `Agent花名册.id` |
| `员工花名` | 文本 | ✅ | 冗余字段，方便看 |
| `任务类型` | 单选 | ✅ | 信息采集 / 爆点分析 / 选题生成 / 文案撰写 / 视觉设计 / 脚本撰写 / 内容审核 / 分发规划 / 数据复盘 |
| `开始时间` | 日期时间 | ✅ | |
| `结束时间` | 日期时间 | ✅ | |
| `耗时（秒）` | 数字 | ✅ | 自动计算 |
| `输入引用` | 长文本 |  | 上游产出的 ID 列表（如 `["TREND-20260503-001", ...]`） |
| `输出引用` | 长文本 | ✅ | 本次产出的 ID 列表（如 `["TOPIC-20260503-005"]`） |
| `日报正文` | 长文本 | ✅ | 字节风格日报，详见 4.3 |
| `异常 / 风险` | 长文本 |  | 有则填，无则空 |
| `下一步` | 长文本 |  | 待办或建议 |
| `状态` | 单选 | ✅ | 成功 / 部分成功 / 失败 |

### 4.3 日报正文格式（字节风格）

每条日志的 `日报正文` 字段使用统一模板：

```
【{花名} · 工作日报 · {YYYY-MM-DD}】

今日完成：
• {时间段} {具体动作}（耗时 {x}s）
• {时间段} {具体动作}（耗时 {x}s）
• ...

产出：
• 共写入 {表名} {N} 条 → {ID 区间}
• {关键质量指标}

异常 / 风险：
• {若无则填"无"}

下一步：
• {交接给谁，做什么}
```

### 4.4 日报示例

```
【小哨 · 工作日报 · 2026-05-03】

今日完成：
• 14:23-14:25 监控 arXiv，抓取 12 条 AI 论文（耗时 2.1s）
• 14:25-14:28 监控 Reddit r/LocalLLaMA，抓取 8 条热帖（耗时 3.2s）
• 14:28-14:30 监控 HackerNews AI 板块，抓取 15 条（耗时 1.8s）

产出：
• 共写入热帖池 35 条 → TREND-20260503-001 ~ TREND-20260503-035
• 高优先级（>0.8）共 7 条

异常 / 风险：
• Reddit API 触发了 1 次速率限制，已自动重试

下一步：
• 等待小析进行爆点分析
```

## 五、Agent 工作日报的写入时机

每个 Agent 在 LangGraph 节点函数末尾，自动写入一条日志：

```python
# 伪代码
def trend_scout_node(state: State) -> State:
    start = datetime.now()
    
    # 干活
    posts = scout.fetch_all()
    output_ids = [storage.create("热帖池", post) for post in posts]
    
    end = datetime.now()
    
    # 自动写日志
    storage.create("Agent协作日志", {
        "id": generate_log_id(),
        "员工 ID": "EMP-001",
        "员工花名": "小哨",
        "任务类型": "信息采集",
        "开始时间": start,
        "结束时间": end,
        "耗时（秒）": (end - start).seconds,
        "输入引用": [],
        "输出引用": output_ids,
        "日报正文": render_daily_report(...),  # 模板渲染
        "状态": "成功",
        "下一步": "等待小析进行爆点分析",
    })
    
    return state.update(...)
```

这个写日志的逻辑可以做成 **decorator**，每个 Agent 节点函数挂上 `@log_work` 自动处理。

def bootstrap_employees():

    employees = load_yaml("seed/employees.yaml")  # 9 条
