# Tables Schema · v3

> v3 架构的 7 张多维表格 schema 定义。
>
> v3 核心理念：**Bitable-Centric 混合架构** —— 状态与元数据存 Bitable，长文本产物存飞书云文档，Bitable 留存文档链接。

---

## v3 核心准则

| 决策项 | 内容 |
|--------|------|
| **状态存储** | 选题状态、资产状态、审改轮次等业务流转全部在 Bitable 表中 |
| **产物存储** | 长文本内容（帖子、脚本、审改记录等）创建为飞书云文档，Bitable 保存文档链接 |
| **API 范围** | Bitable CRUD + 飞书 Docx 创建/追加/读取 + 文档权限设置 |
| **评审体验** | Bitable 看状态，点击链接查看完整内容产物 |

---

## v2 关键变更

| 维度 | v1 | v2 |
|---|---|---|
| 表数量 | 8 张业务 + 3 张元数据 = 11 张 | 4 张业务 + 2 张元数据 = **6 张** |
| 删除的表 | — | 爆点洞察、内容稿件、审查记录、分发计划 |
| 改动的表 | — | 选题库新增"文档链接"+"状态"字段，承载更多职责 |
| 数据回流 | — | 改名为"数据库"，承载追踪 + 经验链接 |

**理由**：v1 把所有产出都往表里塞，v2 保持这种做法——**所有产物全部存在 Bitable 多行文本字段中**，不创建任何飞书云文档。

| # | 表名 | ID 前缀 | 类型 | 谁写 | 谁读 |
|---|---|---|---|---|---|

## 表 1 · 信源配置 `SRC`

### 用途
存储小哨监控的信息源配置。bootstrap 时写入种子数据，运行时小哨读取后调用对应爬虫。

### 字段定义

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `id` | 文本 | ✅ | 业务 ID，例 `SRC-20260504-001` |
| `信源名称` | 文本 | ✅ | 例 "arXiv cs.AI" |
| `平台` | 单选 | ✅ | arXiv / HackerNews / GitHub / Reddit / 小红书 / 抖音 / X |
| `类型` | 单选 | ✅ | 真实爬虫 / Mock 数据 |
| `配置 JSON` | 多行文本 | ✅ | 平台特定配置，例 `{"category": "cs.AI", "limit": 5}` |
| `每次抓取上限` | 数字 | ✅ | 单次最多抓多少条（控制 token 消耗） |
| `是否启用` | 复选框 | ✅ | 关闭可临时禁用某源 |
| `优先级` | 数字 | ✅ | 1-10，小哨按优先级抓取 |
| `创建时间` | 日期时间 | ✅ | 自动 |

## 表 3 · 选题库 `TOPIC` ⭐ 核心枢纽

### 用途
**整个系统的中央调度台**。小编决策、生产组产出、治理组审改、分发结果都通过这张表关联。这张表的状态机是整条流水线的"心跳"。

### 字段定义

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `id` | 文本 | ✅ | 业务 ID，例 `TOPIC-20260504-001` |
| `选题标题` | 文本 | ✅ | 一句话选题 |
| `选题角度` | 多行文本 | ✅ | 200 字内说明从什么角度切入 |
| `预估爆点` | 多行文本 | ✅ | LLM 分析：为什么这个选题会爆 |
| `预估受众` | 多行文本 | ✅ | 目标受众分析 |
| **关联热帖 IDs** | 多行文本 | ✅ | JSON 数组，例 `["TREND-20260504-003"]` |
| `KOC 人设 ID` | 文本 | ✅ | 引用 `KOC.id`，默认 `KOC-001` |
| `推荐优先级` | 数字 | ✅ | 1-10，小编按这个排序 |
| `状态` ⭐ | 单选 | ✅ | 待选 / 已选 / 生产中 / 审改中 / 待发布 / 已发布 / 已弃 |
| `生产开始时间` | 日期时间 |  | 状态变"生产中"时填 |
| `生产完成时间` | 日期时间 |  | 状态变"审改中"时填 |
| `发布完成时间` | 日期时间 |  | 状态变"已发布"时填 |
| **帖子内容** | 多行文本 |  | 小文产出后回填，4平台版本完整内容 |
| **视频脚本内容** | 多行文本 |  | 小播产出后回填，抖音+B站脚本 |
| **审改记录** | 多行文本 |  | 小审/小改累积追加，每轮审查+修改内容 |
| `审改轮次` | 数字 | ✅ | 默认 0，每轮 +1，max 3 |
| `审改最终状态` | 单选 |  | 通过 / 卡死（达到上限未通过） / 待人工 |
| `分发计划 JSON` | 多行文本 |  | 小发产出，存平台×时间×版本，例见下方 |
| `数据回流 ID` | 文本 |  | 引用 `DATA.id`，发布后小数填 |
| `创建时间` | 日期时间 | ✅ | 自动 |
| `创建者 Agent` | 单选 | ✅ | 默认 "小编 TopicCurator" |

### 状态机流转图

```
待选 ──(KOC/小编选定)──→ 已选
                              │
                          (生产组启动)
                              ↓
                          生产中
                              │
                       (生产完成 = 文档链接回填)
                              ↓
                          审改中  ←─────┐
                              │         │
                          (小审审 + 小改改) 循环
                              │         │
                       (通过 OR 达到上限) │
                              │         │
                              └─不通过──┘
                              │
                       (通过 → 小发)
                              ↓
                          待发布
                              │
                       (mock 发布)
                              ↓
                          已发布
                              │
                       (持续追踪)
                              ↓

```

## 我替你拍的几个决策（你不同意可以推翻）

### Flag 1：取消独立"分发计划"表，压缩为选题库的 JSON 字段
**理由**：分发计划是"绑定到选题"的元数据，不是独立实体。每个选题 1 个分发计划，1:1 关系，没必要独立成表。**压缩到选题库的 `分发计划 JSON` 字段**，Agent 读取直接解析 JSON。

**反对理由（可能存在）**：评委可能希望看到"分发计划"作为一张可视化的表（看板视图按平台分组）。

**如果你想保留独立表**：增加第 8 张表 `DIST 分发计划`，字段：选题 ID + 平台 + 时间 + 文案变体 + 受众标签。我倾向不加。

### Flag 2：数据库表把所有平台的字段合并在 1 行
**理由**：1 个选题对应 1 行数据，方便横向对比"这个选题在 4 个平台的表现"。**冗余字段多了点**（4 个平台 × 3-4 个指标 = 12-16 个字段），但可读性最强。

**替代方案**：每个平台 1 行，1 个选题对应 4 行。但这样查询时需要 GROUP BY，飞书 Base 操作不方便。

我倾向**合并 1 行**。

### Flag 3：KOC 人设和 Agent 花名册 ID 不带日期
**理由**：它们是**长生命周期实体**（不会每天新增），ID 带日期反而冗余。

### Flag 4：信源配置作为表而非配置文件
**理由**：让评委看到飞书 Base 里就**直接展示了我们的信源策略**——4 真 + 3 mock 一目了然，"信源策略"本身就是产品价值的一部分。
