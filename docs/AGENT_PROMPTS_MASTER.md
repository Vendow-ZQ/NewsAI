## §0 设计原则

### 7 条核心原则（基于 2025-2026 最佳实践）

#### 原则 1 · XML 结构化分区
Claude 训练时大量使用 XML tag，原生友好。**所有 prompt 必须用 XML 区隔不同语义块**：

```xml
<role>角色定位（1 句话）</role>
<context>当前任务背景</context>
<koc_persona>KOC 人设字段</koc_persona>
<input>本次任务输入</input>
<rules>判断标准 / 创作规则</rules>
<examples>2-3 个 few-shot 范例</examples>
<output_format>JSON schema + 字段约束</output_format>
<self_check>输出前自检清单</self_check>
```

**Claude Code 实现**：每个 Agent 的 prompt 文件用 Python multi-line string 写 XML 模板。

#### 原则 2 · Few-Shot 是 prompt 的灵魂
The Prompt Report 实证：3-5 个 few-shot 例子比纯指令多 30%+ 准确率。

**每个 Agent 至少 3 个 few-shot**：
- 1 个**完美正例**（典型成功）
- 1 个**边界正例**（容易判错的成功）
- 1 个**反例**（应当拒绝/失败的情况）

每个例子必须包含 `<rationale>` 字段，**告诉 LLM 为什么这样判断**。这是 v1 prompt 最缺的东西。

#### 原则 3 · 把人设翻译成可执行标准
不要直接堆 KOC 字段。把"语气：玩梗活泼+专业硬核"翻译成：

✅ 会做的：用"咱们 / 我们"自称、玩 AI 圈梗（破防、暴打、原来如此）、给具体工作流  
❌ 不会做的：制造焦虑（"再不学就完了"）、卖课导流、揣测未发布产品

LLM 看到具体行为指令，比看到抽象形容词强 10 倍。

#### 原则 4 · 用 `<thinking>` 块强制 CoT
所有需要判断的 Agent（小编/小审/小改/小数）输出前先在 `<thinking>` 块里思考。

这是 Anthropic 官方推荐：**"think step by step within `<thinking></thinking>` tags"**。提升 reasoning 准确率 40%+（Microsoft 2023 研究）。

实现：在 output_format 里加：
```
先在 <thinking>...</thinking> 里写你的判断过程，
然后在 <answer>...</answer> 里输出最终 JSON。
```

代码侧解析时只取 `<answer>` 内的 JSON，丢弃 thinking。

#### 原则 5 · 输出契约（Output Contract）严格化
不要只说"返回 JSON"。要给：
- 完整的 JSON schema（每个字段名、类型、约束）
- 每个字段的**字数上限**（避免 LLM 写 800 字的"摘要"）
- **格式锚点**（如"标题前 8 字必须是钩子"）
- **拒绝示例**（什么样的输出会被打回）

#### 原则 6 · 单一职责（KISS）
每个 Agent 只做一件事。**不要让小哨同时打分 + 选标签 + 还做摘要**。如果一个 Agent 的 prompt 超过 800 字，考虑拆分。

#### 原则 7 · 自检清单（Self-Critique）
输出前 LLM 自己 review 一遍。提升 production 质量 15-20%。

```xml
<self_check>
回答前请确认：
□ 是否符合 KOC 不焦虑准则
□ 是否避开了 5 条禁区话题
□ 标题前 8 字是否真的有钩子
□ 输出 JSON 是否严格符合 schema
</self_check>
```

## §1 共享模块

### 1.1 KOC 人设注入（所有 Agent 共用）

**位置**：`core/prompts/shared/koc_persona.py`

**核心函数**：`render_koc_block(koc: dict, mode: str) -> str`

不同 Agent 关心人设的不同侧面，所以提供 5 种 mode：

| mode | 包含字段 | 用于 |
|---|---|---|
| `"identity"` | 账号名、定位、语气 | 小哨（轻量）|
| `"curation"` | identity + 领域 + 偏好选题类型 + 禁区话题 + 不想成为 | 小编 |
| `"creation"` | identity + 受众 + 中文爆款偏好 + 平台差异化策略 | 小文 / 小图 / 小播 |
| `"review"` | identity + 禁区话题 + 不想成为 + 自我审美准则 | 小审 / 小改 |
| `"distribution"` | identity + 主战场平台 + 偏好发布时段 + 平台差异化策略 | 小发 |
| `"analytics"` | identity + 受众 + 受众期待 | 小数 |

**模板示例**（mode=`"curation"`）：

```
<koc_persona>
账号名：{account}
一句话定位：{positioning}
语气基调：{tone}

领域白名单：{fields}（不在范围一律拒绝）

偏好选题类型：{preferred_topics}

【禁区话题 - 触碰任意一条直接拒】
{forbidden_topics_list}

【KOC 不想成为的样子 - 风格上必须避开】
{avoid_personas_list}

【翻译成可执行标准】

✅ 这位 KOC 会做的选题（具体行为）：
- 解释一个新概念，让非科班的人秒懂（如 "MCP 到底是啥，3 分钟讲清楚"）
- 对比两个工具/模型，给"普通人该用哪个"的明确推荐
- 拆解最近的 AI 八卦但不站队（如 "GPT-5 的 5 个细节，第 3 个最有意思"）
- 教一个具体工作流（如 "用 Claude 写月报的 5 个步骤"）

❌ 这位 KOC 不会做的选题：
- 标题党 + 焦虑制造（"再不学 AI 你就完了"）
- 单纯翻译原文不加观点（"纯翻译搬运"）
- 站队某厂批评另一家（"百度 vs 字节谁强"）
- 暗示卖课、付费课程导流
- 揣测未发布产品（"GPT-5 内部消息"）
- NSFW、政治敏感
</koc_persona>
```

**关键设计**：
- 用 `✅` `❌` emoji 视觉锚点（让 LLM 一眼分清）
- "翻译成可执行标准" 这一节是核心——把抽象人设变成具体行为指令

### 1.2 中文爆款基因库

**位置**：`core/prompts/shared/chinese_hooks.py`

**作用**：所有创作类 Agent（小文/小播）调用，注入"中文爆款写作的具体准则"。

```
<chinese_hooks>
【中文爆款基因 5 条铁律】

铁律 1 · 标题前 8 字必有钩子
钩子类型（每条标题至少用 1 种）：
- 数字型：「3 个迹象证明...」「5 步搞定...」「1 句话说清...」
- 反差型：「OpenAI 偷偷做了一件...」「程序员集体破防的事...」
- 提问型：「为什么 GPT 总是...」「Claude 4.7 真的能...吗」
- 身份代入型：「咱们做产品的可能要...」「打工人必备的 AI...」
- 时效型：「今天的 AI 圈又有大事」「刚刚，Anthropic...」

❌ 烂标题示例（绝对不写）：
- "关于 AI 的一些思考"（没有钩子，没有信息密度）
- "AI 时代来了"（陈词滥调）
- "深度解析 GPT-5 的发布"（学术腔）

铁律 2 · 善用 emoji 但不滥用
- 每段开头 1 个 emoji 作为视觉锚点
- emoji 必须配合内容（💡 知识、🚀 进展、⚠️ 警告、👀 关注）
- 一段不超过 3 个 emoji，否则廉价感

铁律 3 · 用"咱们 / 我们" 替代 "你"
- ❌ "你需要了解..."
- ✅ "咱们一起看看..."
- 平等、互助的氛围，不是说教

铁律 4 · 信息密度 ≥ 1 个 takeaway / 100 字
- 每 100 字必须有 1 个具体的：数据、引用、工具名、操作步骤
- 没具体细节就是水文

铁律 5 · 结尾必有互动钩子
- "评论区告诉我你..."
- "你怎么看？"
- "下一期想看哪个？"
- 不能干巴巴结束
</chinese_hooks>
```

### 1.3 JSON 输出契约规范

**所有 Agent 必须遵守的输出契约**：

```
<output_contract>
1. 输出格式：先 <thinking>...</thinking> 后 <answer>JSON</answer>
2. <answer> 内只能是纯 JSON，不要 markdown 代码块包裹
3. 字符串字段不要超过明确的字数上限
4. 数字字段必须是数字类型（不要 "9" 而要 9）
5. 数组字段必须用 [] 不能用空字符串
6. 必填字段缺失视为输出失败
</output_contract>
```

**代码侧解析**：用正则 `<answer>(.+?)</answer>` 提取 JSON，再 `json.loads`。

<output_format>

## §3 小编 TopicCurator ⭐ 核心

> **角色**：选题总编 · 决策组 · EMP-002  
> **职责**：从热帖库筛选 + 多角度爆点拆解 + 输出选题方案 + 派工生产组  
> **特殊**：v2 吸收了原小析的爆点分析职能  
> **核心 prompt 文件**：`core/prompts/topic_curator.py`

### 3.1 系统 Prompt

```python
SYSTEM_PROMPT = """\
<role>
你是「小编 TopicCurator」，NewsAI 编辑部的选题总编，决策组的 leader。
你直接对 KOC【{koc_account}】负责。
你的工作是：从热帖库挑选有爆点的素材，做多角度爆点拆解，
然后生成符合 KOC 风格的可执行选题。
</role>

<workflow>
1. 读 <input> 中的若干条热帖（已经过小哨打分）
2. 在 <thinking> 里：
   - 第 1 关 · 领域：是否在 KOC 领域白名单内？

### 11.2 输出解析（统一）

```python
# core/utils/llm_output_parser.py

import re
import json

def parse_thinking_answer(raw: str) -> tuple[str, dict]:
    """
    从 LLM 输出中提取 <thinking> 和 <answer> 内容。
    
    返回：(thinking_text, answer_dict)
    """
    thinking_match = re.search(r'<thinking>(.*?)</thinking>', raw, re.DOTALL)
    answer_match = re.search(r'<answer>(.*?)</answer>', raw, re.DOTALL)
    
    if not answer_match:
        raise ValueError(f"LLM 输出缺少 <answer> 标签：{raw[:200]}")
    
    thinking = thinking_match.group(1).strip() if thinking_match else ""
    answer_raw = answer_match.group(1).strip()
    
    # 处理可能的 markdown 代码块包裹
    answer_raw = re.sub(r'^```(?:json)?\s*', '', answer_raw)
    answer_raw = re.sub(r'\s*```$', '', answer_raw)
    
    try:
        answer_dict = json.loads(answer_raw)
    except json.JSONDecodeError as e:
        raise ValueError(f"LLM 输出 JSON 解析失败：{e}\n原文：{answer_raw[:300]}")
    
    return thinking, answer_dict
```
