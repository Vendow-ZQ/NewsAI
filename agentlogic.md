● 以下是 9 个 Agent 的完整操作逻辑分析。先说明整体框架，再逐一展开。

  ---
  整体架构

  所有 Agent 继承 BaseAgent，遵循模板方法模式：

  execute() → _read_upstream() → _invoke_tools() → _invoke_llm() → _write_storage() → _log_work()

  LangGraph 编排（core/graph/builder.py）：

  小哨 → 小编 → [Fan-out] 小文 ═╦═ 小图 → [Fan-in] 小审 →{条件边}→ 小改 → 小审(循环) → 小发 → 小数 → END
                                ║
                                ╚═ 小播

  ---
  1. 小哨 TrendScout（信息采集）

  步骤: 读取源
  详情: 信源配置表，过滤 是否启用=true，最多 100 条
  ────────────────────────────────────────
  步骤: 工具调用
  详情: 读取 mock_data/*.json（xiaohongshu_hot.json / douyin_hot.json / github_trending.json /
    hackernews_hot.json / arxiv_papers.json / reddit_posts.json / x_hot.json），每个文件最多取前 2 条
  ────────────────────────────────────────
  步骤: LLM 作用
  详情: 批量打标签+评分。一次性把所有热帖传给 LLM（1 次 API 调用），要求返回 JSON 数组：[{热度评分,
    内容质量, 主题标签}]。如果 mock 数据已有主题标签，则走快速模式跳过 LLM
  ────────────────────────────────────────
  步骤: LLM 超时风险
  详情: 低。单次批量调用，输入是摘要（每条 50 字），输出是短 JSON 数组。约 2-8 秒
  ────────────────────────────────────────
  步骤: 写入目标
  详情: 热帖库表，业务 ID 格式
    TREND-YYYYMMDD-NNN。字段：信源ID/平台/标题/原文链接/摘要/语言/主题标签/阅览量/互动量/发布时间/抓取时
  间/热度评分/内容质量/状态="待选"
  ────────────────────────────────────────
  步骤: 怎么开始
  详情: LangGraph 入口节点（set_entry_point("小哨")），无前置依赖
  ────────────────────────────────────────
  步骤: 怎么停止
  详情: 写完热帖库后自动返回，触发 小编 节点

  关键代码路径：core/agents/trend_scout.py:36-294

  2. 小编 TopicCurator（选题策划）

  步骤: 读取源
  详情: 热帖库表，过滤 状态="待选"，最多 50 条 → 取前 8 条分析
  ────────────────────────────────────────
  步骤: 工具调用
  详情: 无
  ────────────────────────────────────────
  步骤: LLM 作用
  详情: 从 8 条热帖中选出最佳 1 条。Prompt 包含完整的 KOC 人设信息（账号名/定位/目标受众/禁区话题）。LLM

    做"3 关筛查 + 5 维度爆点拆解"思考后，返回 JSON：{最佳选题: {是否适合, 热帖序号, 选题标题, 选题角度,
    预估爆点, 预估受众, 推荐优先级}}。优先级 8 分以上才入选
  ────────────────────────────────────────
  步骤: 判断逻辑
  详情: 如果 selected.get("是否适合")=false 或 推荐优先级<8，则返回空列表，流程中断
  ────────────────────────────────────────
  步骤: LLM 超时风险
  详情: 中。8 条热帖作为 context，LLM 需要比较和判断。约 5-15 秒
  ────────────────────────────────────────
  步骤: 写入目标
  详情: 选题库表，业务 ID 格式
    TOPIC-YYYYMMDD-NNN。字段：选题标题/选题角度/预估爆点/预估受众/关联热帖IDs/KOC人设ID/推荐优先级/状态=
  "已选"/审改轮次=0/创建时间/创建者Agent
  ────────────────────────────────────────
  步骤: 怎么开始
  详情: 小哨完成后顺序触发
  ────────────────────────────────────────
  步骤: 怎么停止
  详情: 写入选题库后返回，触发 Fan-out：同时启动 小文 + 小图 + 小播

  关键代码路径：core/agents/topic_curator.py:92-350

  3. 小文 ContentWriter（文字编辑）

  步骤: 读取源
  详情: 选题库表，过滤 状态="已选"，最多 10 条
  ────────────────────────────────────────
  步骤: 工具调用
  详情: 无
  ────────────────────────────────────────
  步骤: LLM 作用
  详情: 为每个选题生成 4 平台版本（公众号 1500-3000 字 / 小红书 300-500 字 / 抖音 30-60 秒 /
    B站教程）。Prompt 包含 KOC 语气（"玩梗活泼+专业硬核"）和平台差异化策略。返回大
    JSON，含每个平台的标题/正文/标签/CTA
  ────────────────────────────────────────
  步骤: LLM 超时风险
  详情: 高 ⚠️。4 平台长文生成，输出 token
    量大。timeout=120s，可能接近上限。如果有多个选题，会串行循环调用（每个选题 1 次 LLM）
  ────────────────────────────────────────
  步骤: 写入目标
  详情: ① 创建飞书云文档（FeishuDocStorage.create_post_doc()），Markdown 格式写入 4 平台内容；② 更新
    选题库：帖子文档链接、状态="生产中"、生产开始时间
  ────────────────────────────────────────
  步骤: 怎么开始
  详情: 小编完成后，Fan-out 并行触发（与小图、小播同时启动）
  ────────────────────────────────────────
  步骤: 怎么停止
  详情: 写完文档后返回。LangGraph Fan-in 等待小文+小图+小播全部完成后，触发 小审

  关键代码路径：core/agents/content_writer.py:82-412

  6. 小审 Reviewer（审核员）

  步骤: 读取源
  详情: 选题库表，读取 状态="生产中" 或 "审改中"
    的选题；KOC人设表（禁区话题/不想成为的样子/自我审美准则）
  ────────────────────────────────────────
  步骤: 工具调用
  详情: 通过 FeishuDocStorage 读取帖子文档和视频脚本文档内容（各前 2000 字）
  ────────────────────────────────────────
  步骤: 防重复逻辑
  详情: _is_already_reviewed()：如果 状态="审改中" 且
    审改轮次>0，说明已审查过、等待小改修改，跳过不审（防止无限循环）
  ────────────────────────────────────────
  步骤: LLM 作用
  详情: 4 维度审查：事实核查 / 风险词扫描 / 人设一致性 / 平台合规性。返回 JSON：{审查结论:
    "通过"|"需修改", 严重度, 发现的问题: [{位置, 问题, 建议}], 审查指标}
  ────────────────────────────────────────
  步骤: 强制通过机制
  详情: 第 3 轮审查强制通过：if current_round >= 3，自动改结论为"通过"，清空问题列表
  ────────────────────────────────────────
  步骤: LLM 超时风险
  详情: 中。审查相对快，但需要读取云文档内容。约 10-20 秒
  ────────────────────────────────────────
  步骤: 写入目标
  详情: ① 创建/追加审改云文档（v0 原稿 + 每轮审查记录）；② 更新
    选题库：审改文档链接、审改轮次、状态（通过→"待发布"，需修改→"审改中"）
  ────────────────────────────────────────
  步骤: 怎么开始
  详情: Fan-in：小文+小图+小播全部完成后触发
  ────────────────────────────────────────
  步骤: 怎么停止
  详情: 条件边 should_continue_review()：通过 → 触发 小发；需修改 → 触发 小改 →
    小改完成后再触发小审（循环，最多 3 轮）

  关键代码路径：core/agents/reviewer.py:84-770 / core/graph/edges.py:6-17

    设计修改方案。返回 JSON：{修改总结, 修改后的帖子内容: {4平台}, 修改后的视频脚本: {抖音/B站},

  ────────────────────────────────────────
  步骤: 怎么开始
  详情: 小审判定"需修改"时，条件边触发
  ────────────────────────────────────────
  步骤: 怎么停止
  详情: 返回后自动触发 小审（再审）。循环最多 3 轮后强制通过

  小编详细流程展开（你要求的示例）

  ┌─────────────────────────────────────────────────────────────────────────────┐
  │ 小编 TopicCurator 完整执行流程                                               │
  └─────────────────────────────────────────────────────────────────────────────┘

  【Step 1: 开始】
    ↓ LangGraph 边：小哨节点返回后，顺序触发小编节点
    ↓ nodes.py: create_topic_curator_node() 调用 agent.execute(_agent_context(state))

  【Step 2: _read_upstream() — 读取热帖库】
    ↓ 读取源：self.storage.query("热帖库", filters=[状态="待选"], limit=50)
    ↓ 返回：trends 列表（每条含 id/标题/原文摘要/信源平台/热度评分/主题标签/状态）
    ↓ 如果热帖库为空，返回 {"trends": []}，后续流程空转

  【Step 3: _invoke_tools() — 无工具】
    ↓ 小编不需要外部工具，直接返回 {}

  【Step 4: _invoke_llm() — 核心决策】
    ↓ 数据准备：
      - trends_to_analyze = trends[:8]（最多分析 8 条，控制 context 长度）
      - 通过 context.get("koc") 获取 KOC 人设（但当前代码中 context 未传递 koc，
        实际 koc 在 prompt 中使用的是默认值"学AI的刘同学"）
    ↓ Prompt 构建（_build_batch_analysis_prompt）：
      ```
      你是【小编 TopicCurator】，为KOC【学AI的刘同学】选题。
      KOC定位：给非科班大众看的AI资讯速递
      目标受众：22-35岁非技术背景知识工作者
      禁区：政治敏感/引战/卖课/制造焦虑

      候选热帖（8条）：
      [0] 小红书 | AI编程神器Cursor爆火 | score:0.85 | 摘要:Cursor是一款...
      [1] HackerNews | OpenAI发布GPT-5 | score:0.92 | 摘要:OpenAI刚刚...
      ...

      请只选最适合的1条，返回JSON：
      {"最佳选题": {是否适合, 热帖序号, 选题标题, 选题角度, 预估爆点, 预估受众, 推荐优先级}}
      注意：推荐优先级8分以上才值得选。
      ```
    ↓ LLM 调用：self.llm.invoke(prompt)
      - 使用 ChatOpenAI(client.py) → 火山方舟 Doubao 2.0
      - timeout=120s，temperature=0.7
    ↓ 响应解析（_parse_llm_response）：
      - 从 markdown ```json 代码块中提取 JSON
      - 如果解析失败，返回 {"是否适合": False}，流程中断
    ↓ 判断逻辑：
      - selected = analysis.get("最佳选题", {})
      - if not selected 或 not selected.get("是否适合"): 返回空列表，中断
      - idx = selected.get("热帖序号", 0)
      - matched = trends_to_analyze[idx]（通过序号关联回原始热帖）
      - 组装 topic 对象：{热帖ID, 热帖标题, 选题标题, 选题角度, 预估爆点, 预估受众, 推荐优先级}

  【Step 5: _write_storage() — 写入选题库】
    ↓ 生成业务 ID：IDGenerator.generate("TOPIC") → TOPIC-20260511-001
    ↓ 组装记录：
      {
        "id": "TOPIC-20260511-001",
        "选题标题": "Cursor让不会代码的人也能做App？",
        "选题角度": "从'编程门槛消失'角度切入...",
        "预估爆点": "反差感强：程序员 vs 非程序员",
        "预估受众": "22-35岁想提升效率的职场人",
        "关联热帖IDs": "[\"TREND-20260511-003\"]",
        "KOC人设ID": "KOC-001",
        "推荐优先级": 9,
        "状态": "已选",          ← 关键状态
        "审改轮次": 0,           ← 初始轮次
        "创建时间": 1746969600000,
        "创建者Agent": "小编 TopicCurator",
      }
    ↓ 写入：self.storage.create("选题库", record_data)
      → FeishuStorage.create() → FeishuBaseManager.create_record()
      → 通过 IDMapping 维护 business_id ↔ record_id 映射

  【Step 6: _log_work() — 写 Agent 协作日志】
    ↓ 写入 "Agent协作日志" 表：
      {
